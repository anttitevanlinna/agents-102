## Findings

### Azure Foundry Agent Service reached GA (March 16, 2026)
**Source:** https://earezki.com/ai-news/2026-03-18-azure-weekly-foundry-agent-service-hits-ga-and-the-agentic-devops-era-officially-arrives/ — [practitioner analysis]
**Date:** 2026-03-18
**What:** Foundry Agent Service went GA with three headline capabilities: end-to-end private networking (zero public egress for tool calls and MCP connections), stable REST API across Python/.NET/JS/Java SDKs, and Durable Agent Orchestration using Azure Durable Functions for long-running human-in-the-loop workflows via SignalR. The article provides pragmatic use-case guidance (customer support IVR, incident response, sovereign deployments) and warns against specific pitfalls: "high latency and synchronization errors" from manual STT pipelines, and "stateless functions cause data loss during long-wait human interventions."
**Evidence level:** Level 2 (single practitioner analysis of a production-relevant platform, with specific technical warnings)

### Significant technical limitations reported by practitioners
**Source:** https://medium.com/@juliansmiles_40140/azure-ai-foundry-agent-service-technical-limitations-6b0f00ff4adc — [practitioner direct]
**Source:** https://learn.microsoft.com/en-us/answers/questions/5679468/azure-ai-foundry-agent-studio-no-option-for-custom — [practitioner direct, Q&A forum]
**Date:** Jan 2026 / Dec 2025
**What:** Multiple practitioners report blocking limitations: (1) No custom code support within Agent/Workflow service — cannot implement conditional logic or post-processing. (2) Hidden system prompts injected by Azure that users cannot see, control, or override — creates conflict with user instructions. (3) Knowledge source grounding unreliable — agents may ignore connected search indexes and hallucinate. Strict RAG requires search tool workaround, not the default knowledge source. (4) Tools fail randomly on identical parameters. (5) MCP server integration produces generic error messages. These are not edge cases — forum threads show multiple users hitting the same walls.
**Evidence level:** Level 2-3 (multiple independent practitioners reporting same limitations — convergence on the "what doesn't work" side)

### Microsoft's agent ecosystem is sprawling and confusing
**Source:** https://medium.com/@speaktoharisudhan/azure-open-ai-vs-azure-ai-services-vs-azure-ai-foundry-c45e4a7d5013 — [practitioner analysis]
**Source:** https://learn.microsoft.com/en-us/answers/questions/5572779/azure-openai-or-azure-ai-foundry — [practitioner Q&A]
**Date:** Late 2025 / Early 2026
**What:** Practitioners are confused about the relationship between Azure OpenAI Service, Azure AI Foundry, Foundry Agent Service, Copilot Studio, AutoGen, Semantic Kernel, and the Microsoft 365 Agents SDK. Microsoft Q&A threads show developers asking "which one do I use?" — the answer depends on model needs, agent complexity, and existing Microsoft stack investment. Azure OpenAI = direct model access. Foundry = full platform with multi-model. Copilot Studio = low-code for M365. AutoGen = multi-agent framework. Semantic Kernel = lightweight SDK. The surface area is large and the boundaries between products are not crisp.
**Evidence level:** Level 2-3 (convergence — multiple developers expressing the same confusion)

NOTE: This is all vendor marketing — no independent practitioner evidence of Copilot Studio agents running in production for real business processes was found.

### GitHub Copilot coding agent: two modes, real developer traction
**Source:** https://docs.github.com/en/copilot/concepts/agents/coding-agent/about-coding-agent — [vendor docs]
**Source:** https://vocal.media/01/git-hub-copilot-agent-mode-my-honest-review-for-2026 — [practitioner direct]
**Date:** 2026
**What:** GitHub Copilot now has two distinct agent capabilities: (1) Agent Mode in IDE — autonomous editing in local dev environment (create files, run tests, iterate on failures). (2) Coding Agent — cloud-based, assigned via GitHub Issues, creates PRs autonomously. Agent Mode is where most Copilot Pro subscribers spend their time. Can assign issues to Copilot, Claude, or OpenAI Codex agents. Practitioners report it works well for scoped tasks but the "honest review" framing suggests significant caveats exist.
**Evidence level:** Level 2 (individual practitioner reports; this is the coding agent space which is well-trodden, not business agent territory)

### Pricing: pay-as-you-go, no platform fee, but complexity is the cost
**Source:** https://azure.microsoft.com/en-us/pricing/details/microsoft-foundry/ — [vendor docs]
**Source:** https://alrafayglobal.com/azure-ai-foundry-pricing-what-enterprises-need-to-know-before-scaling-ai/ — [practitioner analysis]
**Date:** 2026
**What:** Foundry platform itself is free — you pay for consumed services (model tokens, compute, storage, search). Agent Service managed hosting billing starts no earlier than April 1, 2026 (currently free during preview-to-GA transition). Practitioner analysis warns that real cost drivers at scale are model usage patterns, compute runtime, agent orchestration overhead, and governance — not the platform fee. Model router (GA) can dynamically select cheaper models per prompt to reduce costs.
**Evidence level:** Level 1-2 (pricing facts from vendor; cost structure analysis from practitioner)

## What I Looked For But Did Not Find

1. **Named enterprise deploying Foundry Agent Service agents in production for a specific business process.** The "80,000 enterprises" claim is about Azure AI broadly. No specific "Company X deployed agents via Foundry for Y process with Z results" was found. This is the most important gap.

2. **Independent practitioner comparison of Foundry Agent Service vs. OpenAI Agents SDK.** Given that both can use the same models, the platform differentiation should be in orchestration, security, and tooling — but no head-to-head practitioner analysis was found.

3. **Copilot Studio agents running real business processes.** Lots of marketing about "citizen developer" agent building. Zero evidence of a non-demo Copilot Studio agent handling a real workflow in production.

4. **Pricing at scale.** No practitioner reports of what Foundry Agent Service actually costs at production volume. The "free during preview" period means no one has real cost data yet.

5. **Multi-agent workflow experiences.** Foundry portal now supports visual multi-agent workflow building, but no practitioner reports of actually using this for real work.

6. **AutoGen or Semantic Kernel in production.** These frameworks are part of the Microsoft agent ecosystem but no production deployment reports were found.

## Orientation

Microsoft has the broadest agent platform surface area of any vendor — from low-code (Copilot Studio) to full-stack (Foundry Agent Service) to coding (GitHub Copilot) to frameworks (AutoGen, Semantic Kernel). The GA of Foundry Agent Service (March 2026) with private networking and durable orchestration is the most enterprise-relevant development. However, the practitioner evidence tells a different story than the marketing: real developers report significant limitations (hidden prompts, unreliable grounding, tool failures) and confusion about which of Microsoft's many overlapping products to use. The "80,000 enterprises" number is Azure-broad, not agent-specific. The critical gap: no named enterprise deployment of Foundry agents for a real business process. Microsoft has the distribution channel but the chasm-crossing signal — real enterprises reporting real agent results — has not appeared in public evidence.
