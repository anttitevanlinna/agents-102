### [Finding 2.1] Compound Engineering — Every's "Compound" step as the organizational learning mechanism

**Source:** https://lethain.com/everyinc-compound-engineering/ — [practitioner analysis]
**Date:** 2026 (specific date not confirmed; lethain.com = Will Larson, engineering leader / author)
**Agent level:** team
**What:** Will Larson's analysis of Every's compound engineering methodology names the distinguishing fourth practice that creates compounding vs. linear productivity gains. The four steps: (1) Plan — decoupling research from implementation, consulting specs and PRCs; (2) Work — agentic coding implementation; (3) Review — agents evaluate changes against best-practices; (4) **Compound** — the distinguishing element: "asking the agent to summarize its learnings from a given task into a well-defined, structured format (basically a wiki) which is consulted by future iterations." The critical mechanism: a feedback loop between Compound and Plan stages where knowledge captured from one task automatically informs subsequent planning phases. This is not manual documentation — it's automated organizational learning. Larson's framing: "each completed feature accelerates the next one through accumulated knowledge, reusable patterns, and self-documenting codebases." Larson note on absorption: "Until then, he describes it as 'a cheap, useful experiment that you can implement in an hour.'" These practices will be "absorbed into mainstream AI coding tools like Claude Code and Cursor within months, eventually becoming indistinguishable from folks who are entirely unaware they're using them."
**Evidence level:** L2 (practitioner analysis from named engineering author analyzing a specific named company's method; Every's methodology, not Larson's own deployment)
**Key claims:**
- The "Compound" step is what separates compounding productivity from linear productivity
- Mechanism: structured agent-generated wikis automatically fed into next iteration's planning context
- Without the Compound step, coding agent use produces iteration-by-iteration gains (not compound)
- The feedback loop (Compound → Plan) is the specific mechanism for accumulated organizational knowledge
- Adjacent signal: lethain.com analysis treats this as replicable — low implementation barrier

### [Finding 2.2] Assistants API migration — practitioner-direct post-migration data: 60% faster, 40-60% cost reduction

**Source:** https://medium.com/@gjasula/from-deprecated-to-optimized-a-production-migration-from-openai-assistants-api-to-chat-completions-21d784036644 — [practitioner direct]
**Date:** 2026 (specific month not confirmed; context: migration blog post, pre-August 26 deadline)
**Agent level:** team / company
**What:** Named practitioner (Jürg Steudler) documents a production migration from Assistants API to Chat Completions. Post-migration outcomes: 60% faster responses, 40-60% API cost reduction, code reduced from 450+ lines to streamlined implementation. **Root cause analysis of Assistants API complexity:** thread management required "creating, maintaining, and polling thread states"; multiple async API calls per interaction; model options inflexible (fixed per-assistant, not per-call); "opaque state management on OpenAI's side" made debugging hard. New error patterns differed from prior API. Adopted intelligent model selection per task complexity (gpt-4o for complex, gpt-4o-mini for simpler) — replacing fixed assistant config. Implemented 30-day conversation retention as replacement for thread dependency. Note: This practitioner migrated to Chat Completions, not Responses API — the August 26 migration path is Assistants → Responses API. Different target, but the complexity narrative and outcomes are directly relevant.
**Evidence level:** L2 (named practitioner, specific quantified outcomes, post-migration retrospective; target is Chat Completions not Responses API — label with caveat)
**Key claims:**
- 60% faster responses post-migration
- 40-60% API cost reduction
- Code complexity: 450+ lines → streamlined implementation
- Three primary failure mode categories: (1) context lost across turns; (2) retrieval quality degradation; (3) streaming event pattern breakage
- The "opaque state management" of Assistants was the primary debugging pain point
- Key implication: migrations to Chat Completions completed successfully; Responses API migration is newer and less-documented
