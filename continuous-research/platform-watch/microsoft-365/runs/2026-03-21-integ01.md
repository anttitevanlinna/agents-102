## Findings

### 1. Copilot Studio Connectors: The "1,400 connectors" Reality
**Source:** https://learn.microsoft.com/en-us/microsoft-copilot-studio/advanced-connectors — [vendor documentation]
**Date:** 2026-01-28
**Agent level:** team
**What:** Connectors in Copilot Studio are Power Platform connectors — proxies/wrappers around APIs. They split into Standard (included) and Premium (paid). Each connector exposes specific "tools" (actions) the agent can call. You add tools one at a time to an agent. Connector authentication defaults to user credentials (SSO where supported) or maker-provided credentials. Custom connectors allow connecting to any publicly available API.
**Evidence level:** Level 1 (vendor documentation — describes capability, not deployment)
**Key claims:**
- Connectors are the same Power Platform connectors used in Power Automate/Power Apps
- Premium connectors require paid Copilot Studio plans
- SSO not supported when agent uses custom Active Directory auth + Teams deployment
- No documented limit on how many connectors an agent can use simultaneously
- No documented read-only vs. write permission granularity per connector — permissions follow the connected user's access rights

### 2. Multi-System Orchestration: Architecture Patterns Documented, No Production Evidence
**Source:** https://learn.microsoft.com/en-us/microsoft-copilot-studio/guidance/multi-agent-patterns — [vendor documentation]
**Source:** https://learn.microsoft.com/en-us/microsoft-copilot-studio/guidance/architecture/multi-agent-workflow-oriented — [vendor documentation]
**Date:** 2026 (current docs)
**Agent level:** team / company
**What:** Microsoft documents three multi-agent patterns: inline (child) agents for subroutines sharing context, connected agents for cross-domain delegation, and MCP-based tool integration. Workflow orchestration uses Power Automate, Logic Apps, or Copilot Studio topics. The documentation describes explicit sequencing with preconditions/postconditions, human-in-the-loop gates, and least-privilege connector permissions. However, all examples are architectural — no named enterprise has documented running a multi-system agent workflow in production.
**Evidence level:** Level 1 (vendor architecture docs — describes patterns, zero deployment evidence)
**Key claims:**
- Multi-agent orchestration is GA in Copilot Studio
- Connected agents enable cross-domain delegation (IT agent calling Sales agent)
- MCP support added for standards-based tool integration with enterprise governance (VNet, DLP, auth)
- GPT-4.1 is default model (GPT-4o retired Oct 2025); GPT-5 available in preview
- No production case studies cited in the documentation itself

### 3. Snowflake Integration: Four Patterns, Real Technical Depth
**Source:** https://medium.com/snowflake/the-ultimate-guide-for-connecting-snowflake-to-copilot-studio-which-pattern-fits-53157e790f43 — [practitioner analysis / vendor blog (Snowflake)]
**Source:** https://www.snowflake.com/en/developers/guides/getting-started-with-microsoft-copilot-studio-and-cortex-agents/ — [vendor documentation]
**Source:** https://learn.microsoft.com/en-us/power-platform/release-plan/2025wave1/microsoft-copilot-studio/add-snowflake-as-knowledge-source — [vendor documentation]
**Date:** 2026
**Agent level:** team
**What:** Four integration patterns exist: (1) Native Connector (no-code, read-only queries), (2) Snowflake-managed MCP Server (medium complexity, Cortex AI features), (3) Cortex REST APIs (high control), (4) Custom MCP Server (full orchestration). Snowflake can serve as knowledge source (read-only grounding) or as tool (flexible delegation). The Snowflake Cortex Agents app is available via AppSource for Teams/M365 deployment.
**Evidence level:** Level 1 (vendor documentation with technical implementation guides)
**Key claims:**
- Native connector is read-only (knowledge grounding)
- Tool-mode allows orchestrator to delegate Snowflake calls
- No evidence of enterprises running Copilot Studio agents that chain Snowflake + other systems in production
- MCP-based approach (via CData or custom) can unify Salesforce + Snowflake under one agent — described but not demonstrated in production

### 5. Entra Agent ID: Serious Security Framework, But Preview Only
**Source:** https://learn.microsoft.com/en-us/entra/agent-id/identity-professional/authorization-agent-id — [vendor documentation]
**Source:** https://learn.microsoft.com/en-us/entra/agent-id/identity-platform/what-is-agent-id — [vendor documentation]
**Date:** 2026-02-26 (last updated)
**Agent level:** company
**What:** Entra Agent ID creates a dedicated identity type for AI agents — distinct from users and apps. Key design: least-privilege by default. High-privilege roles (Global Admin, User Admin, Privileged Role Admin) are BLOCKED for agents. Custom roles cannot be assigned to agents. Agents can get Azure RBAC roles scoped to specific resources (e.g., Key Vault Reader on one vault). Two auth modes: delegated (agent acts on behalf of user, bounded by user's access) and unattended (agent acts autonomously with app-assigned roles). Conditional access, network controls, and prompt injection detection included.
**Evidence level:** Level 1 (vendor documentation — the framework is real and detailed, but it's PREVIEW only, available via Frontier early access)
**Key claims:**
- Agent identities are a NEW identity type in Entra, not repurposed user/app accounts
- High-privilege permissions explicitly blocked: Application.ReadWrite.All, RoleManagement.ReadWrite.All, User.ReadWrite.All, Directory.AccessAsUser.All
- Delegated permissions let agents act within user's scope — this IS the RBAC inheritance model
- Azure RBAC can be scoped per-resource (narrow)
- Agent ID Administrator and Agent ID Developer built-in roles exist
- Requires M365 Copilot license + Frontier access; will be in E7 ($99/user) from May 2026
- **Does NOT extend to external system RBAC** — Entra Agent ID governs Microsoft resources. When an agent connects to Salesforce via connector, Salesforce RBAC depends on the connector's OAuth credentials, not Entra Agent ID

### 6. Connector SSO: Works Within Microsoft, Fragile Across Systems
**Source:** https://learn.microsoft.com/en-us/power-platform/release-plan/2025wave1/microsoft-copilot-studio/use-sso-connectors-agents — [vendor documentation]
**Source:** https://learn.microsoft.com/en-us/microsoft-copilot-studio/configuration-end-user-authentication — [vendor documentation]
**Date:** 2025-2026
**Agent level:** team
**What:** SSO for connectors lets users authenticate to external services without additional sign-in steps — when it works. SSO uses OAuth 2.0 Authorization Code + PKCE. Works well within Entra ID ecosystem (M365, SharePoint, Dynamics). For non-Entra connectors (Salesforce, Snowflake, Jira), SSO depends on the external system's OAuth configuration. Tokens are ephemeral — not persisted by Copilot Studio — so long-running workflows need retry/re-auth logic. SSO breaks in specific configs: custom AD auth + Teams channel.
**Evidence level:** Level 1 (vendor documentation)
**Key claims:**
- SSO supported for Entra ID-based connectors natively
- Non-Entra SSO planned in 2025 Wave 2 release plan (not confirmed GA)
- Tokens are ephemeral — no persistent session for agents
- Long-running cross-system workflows must handle re-authentication
- **No unified identity pass-through** — each external connector manages auth independently

### 7. Cross-System Audit Trail: Purview Covers M365, Not External Systems
**Source:** https://learn.microsoft.com/en-us/purview/audit-copilot — [vendor documentation]
**Source:** https://learn.microsoft.com/en-us/microsoft-copilot-studio/admin-logging-copilot-studio — [vendor documentation]
**Source:** https://blog-en.topedia.com/2026/02/auditing-copilot-agent-changes-with-microsoft-purview/ — [practitioner blog]
**Date:** 2026-02
**Agent level:** company
**What:** Microsoft Purview Unified Audit Log (UAL) captures Copilot Studio agent interactions: who interacted, when, what knowledge sources were accessed, conversation transcripts. Computer-using agents (CUAs) get session replays with screenshots for traceability. But: administrative activities logged ONLY for Copilot Studio agents — not for prebuilt agents, third-party agents, or SharePoint agents. External system actions (Salesforce writes, Snowflake queries) are NOT captured in Purview. Each external system logs its own audit trail independently. **No unified cross-system audit trail exists.**
**Evidence level:** Level 1 (vendor documentation + practitioner analysis)
**Key claims:**
- Purview UAL covers M365 Copilot + Copilot Studio agent interactions
- CUA session replays available for compliance
- Non-Microsoft AI app interactions use pay-as-you-go billing for audit
- External system audit trails (Salesforce, Snowflake) are NOT unified into Purview
- SIEM integration (Sentinel, Splunk) required for cross-system audit consolidation
- Data retained 180 days (standard) or 365 days (E5)

### 8. Copilot Memory: Personal Only, Does Not Extend to Agents

**Evidence level:** Level 1 (vendor documentation + European enterprise guide)

- Work IQ layer adds role/project awareness but again, personal Copilot only

### 9. Power Automate as Orchestration Layer: Real Ceiling for Cross-System
**Source:** https://www.kognitos.com/blog/power-automate-limitations-enterprise-automation/ — [vendor competitor blog — treat as biased but directionally useful]
**Source:** https://learn.microsoft.com/en-us/power-automate/limits-and-config — [vendor documentation]
**Date:** 2026
**Agent level:** team
**What:** Power Automate remains the required orchestration layer for multi-system agent workflows in Copilot Studio. Known limitations: rigid trigger-action logic can't handle complex branching; no conversational exception handling (flow can't pause, ask a human, learn, and continue); non-Microsoft connectors introduce latency, require separate auth management, offer limited functionality vs. native integrations. Agent Flows (new) enable autonomous event-driven automation but introduce licensing complexity and governance challenges. March 2026 update adds Power Platform inventory (unified view of flows) and Object-Centric Process Mining.
**Evidence level:** Level 1 (vendor documentation + competitor analysis — no independent practitioner deployment evidence)
**Key claims:**
- Power Automate handles orchestration but hits ceiling on: exception handling, non-Microsoft system depth, branching complexity
- Agent Flows are new but add licensing complexity
- No "pause, ask human, learn, resume" pattern for exceptions
- New process mining helps visibility but doesn't solve orchestration limits
- Premium connectors required for most non-Microsoft systems (additional cost)

### 10. Connector Read vs. Write: Not Granular, Follows User Permissions
**Source:** https://learn.microsoft.com/en-us/microsoft-copilot-studio/advanced-connectors — [vendor documentation]
**Date:** 2026-01-28
**Agent level:** team
**What:** There is NO documented way to set "read Salesforce but don't write" at the Copilot Studio connector level. Connector permissions follow the connected user's access rights in the target system. If the user's Salesforce account has write access, the agent has write access. Permission trimming means "agent can't exceed user's rights" but there's no way to FURTHER restrict within Copilot Studio. The only control is which specific connector "tools" (actions) you add to the agent — you could add only read-actions and omit write-actions, but this is design-time configuration, not runtime enforcement.
**Evidence level:** Level 1 (vendor documentation)
**Key claims:**
- No per-connector read/write toggle in Copilot Studio
- Permissions inherit from connected user's identity in target system
- Design-time control: choose which actions to expose to the agent
- No runtime permission enforcement layer between agent and connector
- Jira connector explicitly notes: when permissions can't be resolved, access is denied (safe default, but crude)

## What I Looked For But Did Not Find

1. **Any named enterprise running a multi-system agent workflow** (e.g., Copilot Studio agent chaining Salesforce + Snowflake + Slack + SharePoint). Zero found across this cycle plus two prior cycles. Three cycles, zero production deployments.

2. **Unified cross-system audit trail.** Does not exist. Purview covers M365 actions; external systems log independently. SIEM integration is the workaround but requires separate setup.

3. **Agent correction-based learning.** Memory exists for personal Copilot but explicitly does NOT extend to custom agents. Agents are stateless between sessions. No mechanism for "learn from this correction."

4. **Granular per-connector permission control.** No "read-only" toggle for connectors. Permissions follow user identity. Design-time action selection is the only control.

5. **Cross-system SSO pass-through.** SSO works within Entra ecosystem. For external systems, each connector manages OAuth independently. No unified identity federation for agents across Microsoft + Salesforce + Snowflake.

6. **Entra Agent ID governing external system access.** Entra Agent ID covers Microsoft resources only. External system RBAC is the external system's problem.

7. **Practitioner reports of building 5+ connector workflows.** No blog posts, forum threads, or conference talks found describing actual multi-connector agent orchestration in Copilot Studio.

## Orientation

### What this cycle changes about the enterprise reality picture:

**The integration story is architecturally ambitious but operationally thin.**

Microsoft has built impressive infrastructure:
- Entra Agent ID is a thoughtful security framework with least-privilege defaults and blocked high-privilege roles
- Multi-agent orchestration patterns are well-documented
- 1,400+ connectors provide breadth of reach
- Purview audit covers the M365 perimeter

But the gaps are structural, not tactical:

1. **No cross-system identity.** An agent connecting to 5 systems authenticates to each independently. There's no unified "agent identity that Salesforce respects." Entra Agent ID stops at Microsoft's border.

2. **No cross-system audit.** If an agent reads SharePoint, writes Salesforce, and posts to Slack, three separate audit logs exist. Unifying them requires SIEM integration — an additional project, not a built-in capability.

3. **No agent memory for custom agents.** Agents built in Copilot Studio are stateless between sessions. No learning, no correction history, no preference accumulation. The Memory feature is personal-Copilot-only.

4. **No granular permission control.** You can't say "read Salesforce but don't write." You can only choose which actions to expose at design time. Runtime enforcement follows the user's credentials in each system independently.

5. **Power Automate remains the orchestration bottleneck.** Complex cross-system workflows still need Power Automate, which can't handle ambiguous decisions, conversational exception handling, or non-Microsoft system depth.

**The 1,400 connectors number is misleading for agent workflows.** Most connectors are designed for Power Automate point-to-point flows, not multi-step agent orchestration. Having a connector means "we can make an API call" — not "we can orchestrate a workflow across this system." The gap between "connect" and "orchestrate" is the enterprise reality.

**For our training content:** This confirms the "agent platform advisory" value prop. An enterprise evaluating Copilot Studio for multi-system business agents would discover these gaps only after significant investment. The connector count is marketing; the orchestration reality requires deep technical assessment. This is exactly the kind of context that only emerges from agentic analysis — no person has comparative experience across these platforms.

**Net assessment update:** After three research cycles, Microsoft's business agent story remains: strong personal-tier tools (Copilot + Memory), ambitious infrastructure (Entra Agent ID, multi-agent patterns, Agent 365 governance), and zero evidence of production multi-system business agent deployments. The integration reality confirms the announcement-to-deployment gap is structural, not just a timing issue.
