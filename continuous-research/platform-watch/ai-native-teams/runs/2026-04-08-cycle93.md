## What Was Found

### New signal: Addy Osmani published a spec-writing framework (Jan 19, 2026)
**URL:** https://addyosmani.com/blog/good-spec/ — [practitioner direct, Google Chrome engineering]
**Also on O'Reilly:** https://www.oreilly.com/radar/how-to-write-a-good-spec-for-ai-agents/ — [practitioner direct syndication]

**Five principles for good specs:**
1. High-level vision first — let AI expand to detail
2. Structured like a PRD — six areas: commands, testing, project structure, code style, git workflow, boundaries
3. Modular prompts over monolithic context ("curse of instructions" — performance degrades as instruction count grows)
4. Built-in self-checks — three-tier boundary system: "Always do" / "Ask first" / "Never do"
5. Iterative testing and evolution — specs as living documents

**Quality evaluation dimensions (implicit, not a formal rubric):**
- Completeness: all six structural areas present
- Specificity: concrete examples over vague descriptions ("too vague" is the most common failure mode per GitHub analysis of 2,500+ agent config files)
- Clarity of constraints: three-tier boundaries explicitly defined
- Alignment with reality: spec reflects actual tech stack and domain knowledge
- Modularity: task size is digestible, not overwhelming

**Critical gap confirmed:** Osmani provides craft principles but **no manager-facing evaluation rubric, no scoring dimensions, no way to compare two people's spec quality**. The framework tells you what a good spec looks like but not how to assess whether a human is consistently producing them.

### Mollick's implicit assessment criterion (from "Management as AI Superpower")
**URL:** https://www.oneusefulthing.org/p/management-as-ai-superpower — [practitioner direct / academic]

Mollick's closest approach to evaluation criteria: **subject matter expertise IS the assessment tool.** Experts who "know what good looks like" can immediately scope problems, define deliverables, and assess AI output quality. The MBA vs. coder experiment confirmed: people with years of domain training outperformed coders because they could translate intent into structured direction.

**Implicit spec-quality framework from Mollick:**
- "What are we trying to accomplish, and why?"
- "Where are the limits of delegated authority?"
- "What does 'done' look like?"

Still not an evaluation rubric. These are craft principles, not a manager's assessment tool.

### Dan Shipper / Every.to — reusable prompt architecture
**URL:** https://every.to/chain-of-thought/compound-engineering-how-every-codes-with-agents — [practitioner direct]

Shipper's compound engineering describes building a "prompt that writes prompts" — a custom Claude Code command that transforms rough feature ideas into fleshed-out specifications. The evaluation is implicit: does the output spec reliably produce good agent execution?

**Key finding:** Shipper uses the OUTPUT to evaluate the spec (did the agent do the right thing?), not the SPEC itself as an independent artifact. No formal framework for evaluating spec-writer skill as a human competency.

### Unicrew / Building AI-Native Development Teams (Feb 12, 2026)
**URL:** https://unicrew.com/blog/building-ai-native-development-teams/ — [general press, practitioner-aligned]

Only criterion offered: "The better the spec (clear acceptance criteria, edge cases documented, input/output contracts defined), the better the AI-generated output." 

Not a manager evaluation tool.

## Gap Analysis Update

The gap confirmed from Pattern G state remains OPEN and unaddressed. The current state of the art:

| Available | Missing |
|-----------|---------|
| What a good spec looks like (Osmani's 5 principles) | How to evaluate whether a PERSON consistently writes good specs |
| What makes specs fail (too vague, monolithic, no constraints) | Manager-facing rubric to assess spec-writing skill |
| Domain expertise as prerequisite (Mollick) | Interview/hiring criteria for spec-writer role |
| Output-based inference (if agent works, spec was good) | Leading indicators before agent execution |

**The closest thing to an evaluation framework found:** Unicrew's three-element definition ("clear acceptance criteria, edge cases documented, input/output contracts defined") — but framed as craft advice, not evaluation criteria.
