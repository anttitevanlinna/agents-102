## What Zep AI Is / How It Works

### Core Architecture

Zep is a **memory layer service** for AI agents, built on top of **Graphiti** — an open-source (24.4k GitHub stars, 800 commits) temporally-aware knowledge graph engine. The architecture has three tiers:

1. **Episode Subgraph** — raw input data (messages, text, JSON). Non-lossy store. Think of it as "what actually happened."
2. **Semantic Entity Subgraph** — entities and relationships extracted from episodes, resolved against existing graph nodes. Think of it as "what the agent knows."
3. **Community Subgraph** — clusters of strongly connected entities with high-level summaries. Think of it as "themes and patterns."

This mirrors psychological models of human memory (episodic vs. semantic).

### The Temporal Innovation — Bi-Temporal Model

Every edge (fact/relationship) carries four timestamps:
- `t_valid` / `t_invalid` — when the fact was actually true in the real world
- `t'_created` / `t'_expired` — when Zep ingested the data (audit trail)

This enables queries like "what did the agent believe at time X?" and "what changed between time A and time B?" — genuine bi-temporal capability. This is what Lindenberg (cycle 84) theorized about, and Zep actually built it.

### Contradiction Handling

When new edges conflict with existing ones, Graphiti uses an LLM to compare semantically related edges. When contradictions are detected, the system sets `t_invalid` on the old edge to match when the new information became valid. **New information is always prioritized** — the system assumes recent data is more accurate.

Critical observation: This is NOT "forgetting" — it's invalidation. The old information remains in the graph with its validity window. The system can always reconstruct what it believed at any point in time. There is NO mechanism for intentional deletion or selective forgetting (e.g., "forget everything about user X" for privacy compliance). The paper explicitly does not address the unlearning problem.

### Retrieval

Hybrid approach: semantic embeddings + BM25 keyword search + graph traversal. Multiple reranking strategies including Reciprocal Rank Fusion, Maximal Marginal Relevance, graph-based episode-mentions reranker, node distance reranker, and cross-encoder LLM reranking.

### Deployment Options

- **Zep Cloud** (managed SaaS)
- **BYOC** (Bring Your Own Cloud) for enterprise
- **Self-hosted** via Graphiti (open-source core). Supports Neo4j, FalkorDB, Kuzu, Amazon Neptune.
- **MCP Server** — Graphiti MCP v1.0 released, enabling Claude Desktop, Cursor, and other MCP clients to read/write knowledge graphs directly.

### Honest Assessment

The multi-agent/shared knowledge capability exists in the product (shared graphs, multi-tenant FalkorDB) but:

1. **The paper doesn't address it.** The arxiv paper has NO discussion of multi-agent scenarios or shared knowledge. The architecture was designed for single-agent memory from conversations.
2. **No evidence of organizational deployment.** Shared graphs are a product feature, not a validated use case. Nobody has demonstrated that a temporal knowledge graph enables cross-boundary learning at organizational scale.
3. **The B2C focus is a tell.** Zep's customers are B2C personalization companies. They remember user preferences and conversation history — a fundamentally different problem than organizational learning. "The agent remembers that User X prefers Adidas shoes" is not "the organization learns which processes benefit from automation."

**Verdict:** Zep's architecture COULD theoretically support organizational memory, but nobody — including Zep themselves — has demonstrated this. The temporal knowledge graph was built to solve personalization (making agents remember individual users), not organizational learning (making organizations accumulate and connect knowledge across domains and teams).

## Connection to Absorption Bottleneck

**Theory:** If a temporal knowledge graph accumulates verified knowledge, agents don't regenerate from scratch each time — reducing the volume of output that needs human review.

**Reality check:** This theory has merit in principle but no evidence yet. The key questions:

1. **Who verifies what goes into the graph?** Graphiti uses LLMs to extract entities and relationships. LLM extraction errors compound over time. There's no human-in-the-loop verification mechanism described in the paper. For the absorption bottleneck, this means the graph could accumulate garbage as easily as knowledge.

2. **Does it actually reduce human review load?** No evidence. If agents pull from a graph that itself contains LLM-extracted (potentially hallucinated) facts, the human still needs to verify the output. You've moved the verification problem, not solved it.

3. **The contradiction-handling assumption.** Graphiti assumes new information is always more accurate than old. In organizational contexts, this is often wrong — a junior agent's recent output shouldn't automatically override a senior human's validated finding. The system has no concept of authority or confidence levels.

### The Counter-Argument: Is Memory Infrastructure Even Needed?

**Letta's filesystem argument (August 2025):** A simple agent using file operations scored 74% on LoCoMo — competitive with specialized memory systems. Their thesis: agents already know how to use filesystems (it's in their training data). Specialized memory APIs add complexity without proportional benefit.

**Letta's Context Repositories (February 2026):** Git-based memory for coding agents. Files + git versioning + hierarchical structure. No knowledge graph, no vector database. The argument: "Files are simple, universal primitives that both humans and agents can work with using familiar tools."

[practitioner direct] Letta context repos: https://www.letta.com/blog/context-repositories

**The stateless re-derivation argument:** Coding agents (Claude Code, Codex) re-derive context from scratch each session by reading repos, docs, databases. They don't need persistent memory — they re-read the source of truth. This is arguably simpler and more robust than maintaining a knowledge graph that might accumulate errors.

**15-25% context re-establishment tax:** Practitioners report spending 15-25% of interaction time re-establishing context with agents. This is the cost of stateless approaches. Whether a knowledge graph is worth the complexity to eliminate this tax is an open question.

## Orientation

### What This Means for Our Research

**The temporal knowledge graph is technically real and architecturally novel.** Zep's bi-temporal model (validity intervals + audit timestamps) is a genuine innovation. Graphiti's open-source adoption (24.4k stars) shows practitioner interest. The architecture COULD enable the kind of temporal organizational memory our "broad learning" thesis requires.

**But nobody is using it for organizational learning.** The entire agent memory category is solving a different problem — personalization and coding agent continuity. There's a large gap between "temporal knowledge graph for remembering user preferences" and "temporal knowledge graph for organizational learning accumulation." Zep's B2C focus, small team, and limited funding suggest this gap won't be closed by them.

**The counter-argument is strong.** Letta's filesystem-based approach (context repositories, git-backed memory) is simpler, more transparent, and scored competitively on benchmarks. For coding agents specifically, the stateless re-derivation approach (just re-read the repo) might be strictly better than maintaining a knowledge graph — it guarantees you're working from the actual source of truth, not an LLM's extraction of it.

**The benchmark wars destroy credibility for everyone.** Mem0 misconfigures Zep's system. Zep rebuts. Letta says all the benchmarks are wrong. Hindsight comes in and outscores everyone. This is vendor marketing dressed as science. Evidence level for the entire category: 1-2 at best.

### Revised Assessment of Memory Infrastructure for Broad Learning

**Previous hypothesis:** Zep's temporal knowledge graph could be the memory infrastructure that makes compound engineering work at organizational scale.

**Updated hypothesis:** Unlikely in the near term. The technology is pointed at the wrong problem (B2C personalization, not organizational learning). The team is too small and underfunded to pivot. And the simpler alternative (stateless re-derivation by coding agents + git-based memory) may be adequate for most organizational use cases without the complexity and error-accumulation risks of a knowledge graph.

**What would change this assessment:**
1. Evidence of ANY organization using temporal knowledge graphs for cross-team learning (not personalization)
2. A benchmark showing agents with graph memory outperform agents with re-derivation on real enterprise tasks
3. A verification/human-in-the-loop mechanism for knowledge graph content (addressing the error accumulation problem)

### Research Priority

**Lower the priority of agent memory infrastructure.** The category is crowded, benchmark-obsessed, and pointed at personalization — not our domains. Monitor Hindsight (new, highest scores) and Letta's context repositories (simplest, most transparent) as the two most interesting threads. Check back in 3-6 months for organizational deployment evidence.

**Higher priority: the absorption bottleneck itself.** The real question isn't "where do you store agent knowledge?" — it's "who verifies it?" Neither Zep nor any competitor addresses this. The human-in-the-loop verification problem is upstream of the storage problem.
