 - **The problem.** Two M4 prompts touch branch identity and never talk to each other. (1) `walk-and-send-off-1` (the scope prompt the student copies) says only *"…scope it into two or three sentences I can refer back to … save the scope as `./task.md` in the repo root (overwrite if it exists)."* It puts **no constraint on task.md's shape**, so Claude's task-card instinct stamps a `Branch:` header at the **top** of task.md, and does so at scope-time, *before any branch exists*. (2) Later, `ae101-m4-commit-starting-point` says *"Create a fresh feature branch named `m4/<short-task-slug>`. Before committing, append a protected block to `./task.md` headed `Run coordinates (do not rewrite or remove)` … the branch name exactly as `git branch --show-current` reports it (read it back from git, don't retype it from memory) …"*. It hardens the **appended coordinates** line via git read-back, but is silent about the `Branch:` header the earlier step may have left above it. Net: task.md ends with **two `Branch:` lines naming different branches**, and only the coordinates one is authoritative.
 - **Why this is student-facing, not just the test harness.** (a) **The recovery anchor lies to the human.** A student who opens their own `task.md` to resume a run sees two branch names and has no way to know which the tooling trusts. The 2026-05-26 hardening (`df17bc0`) taught the *machine* reader `ae101-m5-worktree-setup` to prefer the `Run coordinates` block and ignore any other body `Branch:` line, but that did nothing for the human reading the doc, and nothing for the authoring step that creates the contradiction. We patched the reader, not the writer. (b) **`ae101-m4-commit-starting-point` assumes the student isn't already on a task branch.** A real engineer often *is* (they branched before starting the task). *"Create a fresh feature branch named `m4/<short-task-slug>`"* then spawns a second, curriculum-named branch and commits the work there, diverging from the branch the student believes the work lives on.
