### Finding 2: Enterprise agent controls tightened — role-based access, domain blocking
**Source:** [ChatGPT Agent Help Center](https://help.openai.com/en/articles/11752874-chatgpt-agent) — [vendor documentation]
**Date:** March 2026
**Agent level:** team / company
**What:** Enterprise workspace owners can now: (1) enable/disable agent mode per workspace (default OFF), (2) assign agent mode to specific roles, (3) block specific websites/domains from agent browsing, (4) control which apps are available. This is the first meaningful RBAC-adjacent capability from OpenAI — not full RBAC, but role-based feature gating.
**Evidence level:** Level 0 (vendor docs — factual capability)
**Key claims:**
- Agent mode defaulted to OFF for enterprise workspaces
- Role-based assignment of agent mode capability
- Domain blocking for agent browsing
- Per-workspace app availability controls

## Orientation

**The picture is shifting from "read-only toy" to "write-capable but unproven."** The connector evolution is significant — OpenAI has moved from read-only to write actions for Google, Microsoft, and Jira. This closes one of the 9 structural gaps identified in previous cycles. However, there's a new gap: write actions exist but nobody has reported using them for real business workflows. The enterprise controls (role-based agent assignment, domain blocking) are the first steps toward the RBAC capability that was completely missing.

**The honest update for the CTO:** OpenAI is building the plumbing faster than expected. Write-action connectors + enterprise controls + recurring scheduling = the ingredients for business agent workflows. But the ingredients aren't a recipe yet. Zero named enterprises are running agentic business processes, and the agent mode's sandboxed virtual computer architecture remains fundamentally disconnected from logged-in business systems.
