* answers: 1) Framework is Go's `testing` package, `go test ./...`. No jest/vitest analog, no third-party test framework. 2) Mocked vs real: very little is mocked. Index tests build a real temp index via t.TempDir() + index.Create() + AddFile() + Flush(), then read it back — see index/write_test.go's buildFlushIndex helper. HTTP tests would use net/http/httptest (recorder for unit, NewServer for integration); csweb has zero tests today, so this is greenfield. No browser, no DOM, no fetch — the page-load JS in cmd/csweb/_static/viewer.js has no test harness in this repo. 3) Integration boundary: the filesystem (the index file at $CSEARCHINDEX) for the search core, and HTTP for csweb. No external service, no third-party SDK. 4) Flaky: rare. Geometry-shaped bugs surface as real bugs not flakiness — see M1's siftUp fix (TestPostHeapSiftUp catches wrong heap output deterministically, never intermittently). The mmap path on a missing/corrupt index calls log.Fatal which kills the test process — that's a real bug surfacing as a process exit, not flakiness. 5) Regression scope: full `go test ./...` runs in well under a second. Cheap to run on every change. 6) Load-bearing tests reviewers check first: index/write_test.go (TestTrivialWrite + the new TestPostHeapSiftUp / TestManyChunkMerge from M1's deeper pass), regexp/regexp_test.go (TestGrep), regexp/match_test.go.
