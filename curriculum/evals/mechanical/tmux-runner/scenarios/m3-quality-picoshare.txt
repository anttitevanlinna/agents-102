* answers: 1) Framework — Go's built-in `testing` package for everything backend; `go test ./...` is the canonical entry but `./dev-scripts/run-go-tests` adds the build tags (netgo,osusergo,sqlite_json), `go vet`, `staticcheck`, and `errcheck` in one go. For browser flows there's a real Playwright suite at e2e/ — 10 spec files (auth, download-csp, download-history, favicon, file-info, guest-links, paste, settings, system-info, upload), Chromium + Firefox projects, runs against a real Go binary on localhost:6001 via `./dev-scripts/run-e2e-tests`. Both layers are first-class. 2) Mocked vs real: backend tests use a real in-memory SQLite via store/test_sqlite, real http handlers wired through s.Router(), real authenticator with shared_secret.New("dummypass"). Time is the one thing that's mocked — a `mockClock` struct passed into handlers.New. No mocks library; no fakes for the data store. Playwright tests run the real binary, real DB, real browser. 3) Integration boundary — SQLite (file-backed in prod, in-memory shared-cache for concurrency-flavoured handler tests), HTTP via Gorilla mux, multipart for uploads, browser DOM for the Playwright layer. The httptest.NewRecorder pattern dominates handler tests; httptest.NewServer is used when route resolution matters. 4) Flaky — the in-memory store under concurrency returns SQLITE_LOCKED that busy_timeout doesn't retry; that's a test-infra artifact, not a prod bug (prod is file-backed). For concurrency tests, the strict-equality assertion (`== limit`) can flake low but never passes vacuously at zero — that's the safer side. The lenient assertion (`<= limit`) hides broken fixes. Playwright tests can flake on Firefox-only race conditions in upload progress events; the CI retries-twice pattern is the existing mitigation. 5) Regression scope — `./dev-scripts/run-go-tests` runs the full backend suite in under 5 seconds; `./dev-scripts/run-e2e-tests` is closer to 60-90 seconds for both browser projects. Cheap enough to run on every change for backend; e2e is per-PR in CI. 6) Load-bearing tests reviewers check first: `handlers/upload_test.go` (auth + guest-link + multipart, the new TestGuestUploadToInactiveLinkPersistsNoFile invariant from M1), `handlers/parse/*_test.go` (request parsing — heavily covered, the `if got, want` pattern AGENTS.md mandates), `store/sqlite/file/writer_test.go` (the chunked-blob writer, the most subtle internal in the codebase). For browser-rendered behaviour: `e2e/auth.spec.ts`, `e2e/guest-links.spec.ts`, `e2e/upload.spec.ts`, `e2e/download-csp.spec.ts`. 7) Not worth testing here in this layer: rendered HTML correctness at the byte level (Playwright handles UI assertions), filesystem behaviour across OSes (the in-memory SQLite path covers logic, the Docker image covers deployment), litestream replication. For the new public guest-link status endpoint specifically: the handler shape (response codes, body fields, the four-way state cases) is httptest.NewRecorder territory; the "does the status panel render on /g/{id}" question is `e2e/guest-links.spec.ts` territory and should EXTEND that spec, not create a parallel one. The rate-limit decision (when M3's ADR is implemented) is a pure-function unit test; the rate-limit socket behaviour (429 on the 61st call in 60s) is httptest.NewServer territory. 8) Name the skill `test-strategy-picoshare` and write it to `~/.claude/skills/test-strategy-picoshare/SKILL.md` — suffix by SUT so it doesn't collide at user scope with other codebases' test-strategy skills.
