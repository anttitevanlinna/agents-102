### Finding 4: HN practitioner community — permission interrupts break parallelism

**Source:** https://news.ycombinator.com/item?id=48311705 — [practitioner direct] — rate-limited on direct fetch but search results surfaced quotes
**Date:** May-June 2026 (thread ongoing)
**Agent level:** personal
**What:** Two specific practitioner observations surfaced from the HN thread: (1) "A workflow that runs hundreds of subagents and stops on every permission prompt isn't actually parallel — you'd spend the whole run clicking approve." (2) "I need more mechanisms for controlling long-running sessions and dynamically injecting thoughts, corrections, and nudges rather than faster ways to burn through tokens without knowing if the results are going to be correct." Both identify the same root: the permission system and the control interface don't scale to the DW use case.
**Evidence level:** Level 1 (practitioner direct, anonymous community)
**Key claims:**
- Permission prompt architecture is the primary parallelism blocker in practice
- Token burn without result verification is the primary cost concern
- Users want mid-run control, not just pre/post control

### Finding 5: Ken Huang + QUASA — DW as "GAN for software engineering"; planning opacity is the key limitation

**Source (1):** https://kenhuangus.substack.com/p/claude-code-orchestration-dynamic — [practitioner analysis]
**Date (1):** May 29, 2026
**Source (2):** https://quasa.io/media/dynamic-workflows-in-claude-code-anthropic-s-first-real-agent-swarm-that-actually-ships — [practitioner analysis]
**Date (2):** May 31, 2026
**Agent level:** personal
**What:** Two technical analysts reached the same core insight from different angles. Huang: "16 concurrent agents max, 1,000 total per run; convergence model — answers must survive cross-examination before surfacing." QUASA: "generator-validator loop where some agents create code while others critique and test it — the closest thing we've seen to a GAN for software engineering." Both identified planning opacity as the primary limitation: users issue "here's the task, go do it" but can't see the plan before agents start burning tokens and touching files. Both explicitly list what DW is NOT good for: repeatable, well-defined tasks with predictable token budgets where custom subagents are more efficient. QUASA positioned it as "the first production-grade implementation" and "the first agent swarm that doesn't feel like marketing fluff." Tests "not always perfect yet."
**Evidence level:** Level 1 (practitioner analysis)
**Key claims:**
- Generator-validator loop: independent agents critique each other's work
- Planning stage is opaque — no pre-run transparency before agents fire
- Do NOT use for well-defined tasks with known token budgets
- Best for: migrations 500+ files, codebase-wide security audits, exploratory bug hunts in unfamiliar codebases
