map-the-access-surface-4 The new GET /api/guest-links/{id}/public-status has no audit logging — but picoshare's existing POST /api/guest/{guestLinkID} (the upload endpoint that pairs with it) calls log.Printf on every save. So uploads to a guest link are observed; status queries against the same link are not. The skill called this read/write audit asymmetry out harder than I would have — I was modeling the status endpoint as additive ("just expose state that's already there") and missed that "observable" and "observed" are different things. Concretely: an attacker probing the status endpoint to enumerate live guest-link IDs leaves no trace in picoshare's logs, while a single upload attempt against the same ID would show up. The picoshare-specific bite is that the existing upload logging is granular and well-instrumented; adding a sibling endpoint without a parallel audit hook erodes the read/write symmetry the rest of the codebase maintains.

map-the-access-surface-5 The response shape itself is an enumeration oracle for the upload endpoint's bearer-token auth. A 200 with the JSON envelope means the guest-link ID resolves to a real record (active or not); a 404 means it doesn't. An attacker can confirm whether a candidate base32 ID is a real guest link without uploading anything — and guest-link IDs are the only auth on POST /api/guest/{id}. The skill catalogs endpoints and verbs as separate surfaces; it does not by default model that a read endpoint's response surface is the enumeration oracle for a sibling write endpoint's auth tokens. Codebase-specific because picoshare's guest-link IDs are 16-character base32 — long enough that random guessing is infeasible, but any oracle that confirms validity collapses the search space and makes targeted abuse plausible.
