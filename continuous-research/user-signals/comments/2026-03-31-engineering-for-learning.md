Practitioner reframe: the engineering contract must change from "build for stability" to "build for learning." Traditional systems engineering optimises for stability, predictability, repeatability. Change is risk. The new contract: systems must learn and improve with use. Change is the product.

The compounding in compound engineering is not about producing more code — it's about the SYSTEM learning. CLAUDE.md gets richer, agents get smarter, context accumulates, each cycle makes the next better. Code is a byproduct of learning.

**Reframes every finding in the system:**
- Verification infrastructure = the feedback loop that enables learning (Level 2 generates data that trains Level 3)
- Absorption bottleneck = a learning speed problem, not just a review problem. If humans can't absorb output, the org stops learning.
- Specs = hypotheses. Output confirms or refutes. Spec evolves. That's learning.
- StrongDM holdout scenarios = a learning architecture, not just verification
- The real moat = organizational learning rate encoded in the engineering system

**Gap:** Nobody frames AI-era engineering as "engineering for learning" vs "engineering for stability." This is a potentially nameable architectural paradigm shift. Who is closest? Karpathy (autoresearch as learning loop)? StrongDM (holdout scenarios as learning architecture)? Compound engineering (CLAUDE.md as accumulated learning)?

**CTO question reframed:** Not "how do we build stable systems with AI?" but "how do we build systems that learn faster than competitors' systems learn?"
