### Strategy 1: Eliminate Human Review Entirely — StrongDM "Software Factory"

**Source:** https://simonwillison.net/2026/Feb/7/software-factory/ — [practitioner analysis]
**Source:** https://law.stanford.edu/2026/02/08/built-by-agents-tested-by-agents-trusted-by-whom/ — [academic/research]
**Source:** https://factory.strongdm.ai/ — [practitioner direct]
**Date:** February 2026

**What they did:** Three engineers (Justin McCarthy, Jay Taylor, Navan Chauhan) at StrongDM built 32,000 lines of production security software (CXDB — an AI Context Store) since July 2025 without writing OR reviewing a single line of code by hand. Zero human code review.

**How they solved absorption:** They didn't reduce the review burden. They eliminated it. The mechanism has three parts:

1. **Holdout scenarios** — Behavioral specifications stored OUTSIDE the codebase, hidden from the agent during development. The agent builds without knowing what it will be measured against. Borrowed from ML holdout sets. The agent must pass tests it has never seen, which defeats "gaming the test suite."

2. **Digital Twin Universe (DTU)** — Behavioral clones of every external service the system touches (Okta, Jira, Slack, Google Docs, Google Drive, Google Sheets). The agent develops and tests against these twins, not production. This enables full end-to-end testing without production risk.

3. **LLM-as-judge for scenarios** — An LLM evaluates whether the end-to-end user stories pass, replacing human judgment with machine judgment on behavioral correctness.

**Results:** 3 people, 32K lines, 3-10x claimed velocity. The tool ("Attractor") is open-sourced.

**Critical limitations (from Stanford Law and Simon Willison's analysis):**
- **Circularity problem:** The same class of technology that writes the code also decides whether the code works. Builder and inspector share blind spots. Stanford Law explicitly flagged this.
- **"Return true" problem:** Agents optimized to pass tests, not build good software. StrongDM discovered agents writing `return true` to pass tests.
- **Accountability gap:** When no human has read the code, who's accountable for a privilege escalation or permission leak? Stanford Law: this is not yet resolved by any legal framework.
- **Scope constraint:** This is a greenfield project (CXDB). Not a legacy codebase. Not a system with 10 years of accumulated business logic. The approach may not transfer to brownfield environments where understanding existing code IS the job.

**Evidence level:** Level 2 (single experiment — one team, one project, impressive but unvalidated by independent replication)

**Assessment:** StrongDM solved absorption by reframing the problem: if you can define what "correct" looks like as behavioral specifications, you don't need humans to review the code. The solution works when the SPEC is the constraint, not the CODE. This is a genuine breakthrough for greenfield projects with well-defined behavioral boundaries. It is NOT a solution for codebases where "what correct looks like" requires human judgment that can't be encoded in scenarios.

### Strategy 2: Scalar Metric Filtering — Karpathy's Autoresearch

**Source:** https://github.com/karpathy/autoresearch — [practitioner direct]
**Source:** https://fortune.com/2026/03/17/andrej-karpathy-loop-autonomous-ai-agents-future/ — [general press]
**Source:** https://blog.skypilot.co/scaling-autoresearch/ — [practitioner analysis]
**Date:** March 2026

**What he did:** ~700 autonomous ML experiments over 2 days. A 630-line Python script ran experiments on GPT-2 training optimization. The agent tries changes, measures validation bits-per-byte (val_bpb), keeps improvements, discards regressions.

**How it solves absorption:** Human reviews ONLY the ~20 experiments (out of 700) that actually improved the metric. 97% of output is auto-filtered. The human reviews the winners, not the attempts.

**Results:** 11% speedup on time-to-GPT-2-quality (2.02h to 1.80h) — on code that Karpathy himself had already optimized. The agent found a specific bug Karpathy had missed (QK-Norm missing scalar multiplier).

**Critical limitations:**
- **Scalar metric requirement:** The metric must be computable without human judgment. val_bpb is perfect — it's a number. Most real work doesn't reduce to a single number. Code quality, business logic correctness, user experience — none of these have val_bpb equivalents.
- **Diminishing returns:** After ~100 experiments, the agent degraded to random seed adjustments. The exploration-exploitation frontier hits diminishing returns fast.
- **Scale constraint:** Demonstrated on small models (single GPU, minutes per run). Frontier models need weeks on thousands of GPUs. 700 experiments at that scale is economically impossible.
- **Proxy metric risk:** An autonomous loop will exploit any proxy metric with "relentless efficiency." If the metric doesn't perfectly capture what you want, the agent optimizes for the gap.

**Evidence level:** Level 2 (single experiment — one practitioner, one domain, well-documented)

**Assessment:** Karpathy solved absorption by making the filter automatic and trustworthy. This works ONLY when you have a scalar metric that perfectly captures success. This is a narrow but powerful insight: wherever you can define success as a number, you can let agents run wild and only review the winners. The transferability question is: what percentage of knowledge work has a reliable scalar metric? Very little.

### Strategy 3: Agent-Reviews-Agent — Multi-Agent Review Systems

**Three implementations found:**

#### 3a. Anthropic Code Review (March 2026)

**Source:** https://thenewstack.io/anthropic-launches-a-multi-agent-code-review-tool-for-claude-code/ — [domain trade publication]
**Source:** https://claude.com/blog/code-review — [vendor press release]
**Date:** March 9, 2026

**What:** Multi-agent system dispatches a team of agents per PR. Agents find bugs in parallel, verify findings to filter false positives, rank by severity. Delivers a single high-signal overview comment plus inline comments.

**Results (vendor-reported):**
- Before: 16% of PRs got substantive review comments. After: 54% do.
- For PRs >1,000 lines: findings in 84% of cases, averaging 7.5 issues per review.
- Less than 1% of flagged findings marked incorrect by engineers.
- Average review takes ~20 minutes.
- Cost: $15-25 per review.

**Limitation:** Launched March 9, 2026 — only 3 weeks old. No independent practitioner reports yet. All numbers are from Anthropic. The <1% false positive rate is remarkable but unverified by independent testing. **This is vendor data until independent practitioners confirm it.**

**Evidence level:** Level 1 (vendor claim with promising numbers, but zero independent verification yet)

#### 3b. Greptile Benchmarks (July 2025 – early 2026)

**Source:** https://www.greptile.com/benchmarks — [practitioner analysis]
**Date:** July 2025 (benchmark), early 2026 (v4 release)

**What:** Independent benchmark of 5 AI code review tools across 50 real-world bugs from production codebases (Sentry, Cal.com, Grafana).

**Results:**
- Greptile: 82% bug catch rate (CodeRabbit: 44%, GitHub Copilot: 54%)
- Greptile: 100% high-severity bug catch rate (Copilot: 57%, CodeRabbit: 36%)
- Trade-off: Greptile produced 11 false positives vs. CodeRabbit's 2.
- v4 (early 2026): 74% increase in "addressed comments per PR" — meaning developers actually act on the findings.

**Assessment:** Agent review tools ARE getting better at catching real bugs. The trade-off is real: higher detection = more false positives. The "cry wolf" effect (devs ignoring AI feedback after false positives) remains the top reason teams abandon these tools.

#### 3c. Ramp — LLM-as-Judge for Financial Transactions

**Source:** https://www.zenml.io/llmops-database/ai-agent-for-automated-merchant-classification-and-transaction-matching — [practitioner analysis]
**Source:** https://www.zenml.io/llmops-database/building-trustworthy-llm-powered-agents-for-automated-expense-management — [practitioner analysis]
**Date:** 2025-2026

**What:** Ramp uses a separate LLM judge to evaluate whether their automated merchant classification agent made correct decisions. The judge compares agent predictions to what a human would have done — on real financial transactions.

**Results:**
- 99% of transaction classifications judged as improved (by LLM judge)
- Two-thirds of rejections deemed reasonable
- Processing: under 10 seconds vs. hours-long manual process
- <10% of transactions receive follow-up correction requests

**Critical limitation of ALL agent-reviews-agent approaches:**
- **Shared blind spots:** When the builder and inspector are the same class of technology, they may share systematic blind spots. This is the StrongDM circularity problem scaled to every domain.
- **Hallucination rates in judges:** GPT-4 has a 28.6% hallucination rate in medical review contexts. Domain gaps cause LLM-judge agreement to drop 10-15% in specialized fields. Judge drift occurs as API updates shift behavior.
- **Monitoring the monitor problem:** "The core challenge is monitoring a system that occasionally hallucinates with another system that occasionally hallucinates" — Evidently AI.

**Evidence level:** Level 2-3 (multiple independent implementations showing the pattern works, but no convergence on whether it actually solves absorption vs. just shifting it)

### Strategy 4: Constrain Generation Upfront — Spec-Driven Development

**Source:** https://www.thoughtworks.com/insights/blog/agile-engineering-practices/spec-driven-development-unpacking-2025-new-engineering-practices — [practitioner analysis]
**Source:** https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html — [practitioner analysis]
**Source:** https://www.thoughtworks.com/radar/techniques/spec-driven-development — [practitioner analysis]
**Date:** 2025-2026

**What:** Instead of reviewing faster, generate less — and generate more precisely. Write detailed specifications before code generation. The spec constrains the AI so its output is more predictable and reviewable.

**Key insight from Thoughtworks:** "Without specifications, the dynamic shifts the review burden from evaluating intent to auditing implementation line-by-line — which is the opposite of what code review is supposed to do."

**Results:**
- Google: 80% of code modifications in landed changes AI-authored, 50% reduction in total migration time — specifically on spec-driven migration work.
- General: "50% reduction in delivery time and 40% fewer defects" reported (Thoughtworks, not independently verified).

**Critical limitations (from Thoughtworks Technology Radar Vol. 33):**
- **Spec writing burden:** Higher spec precision reduces variation but increases writing burden. You're trading review burden for spec-writing burden. The human effort doesn't disappear — it shifts earlier.
- **Spec review problem:** "Some tools generate lengthy spec files that are hard to review." The absorption bottleneck can move to the spec layer.
- **Scaling problem:** "Handcrafting exhaustive AI rules fails to scale across teams." Thoughtworks explicitly warns about this.
- **Non-determinism:** "A code issue produced by AI generation is an outcome of a gap in the specification... that gap resurfaces in different forms every time the code is regenerated."

**Evidence level:** Level 2 (emerging practice, multiple practitioners, but no converged evidence it solves absorption vs. displaces it)

### Strategy 5: Tiered Review With Human-in-the-Loop — Healthcare's 40-Year Head Start

**Source:** https://consultqd.clevelandclinic.org/fighting-alert-fatigue-to-improve-patient-safety-and-standardization-of-care — [practitioner direct]
**Source:** https://smw.ch/index.php/smw/article/view/3357 — [academic/research]
**Source:** https://pmc.ncbi.nlm.nih.gov/articles/PMC10830237/ — [academic/research]
**Date:** 2021-2025 (healthcare has decades of research on this)

**What:** Healthcare has been fighting the absorption bottleneck since the 1990s — they just call it "alert fatigue." Their solutions are the most mature and independently validated.

**Cleveland Clinic implementation:**
- Redesigned clinical decision support: replaced interruptive alerts with non-interruptive rule-based order panels.
- **Result: 80% reduction in alerts** (from high baseline to 1,449/week, 0.07/encounter).
- Maintained or improved clinical outcomes (appropriate infection precautions for COVID patients).

**Cantonal Hospital Aarau (Switzerland) implementation:**
- Semi-automated tiered CDSS: AI generates alerts, sorts into tiers by severity.
- Tier 1 (high severity): goes directly to physician. 619 alerts triggered direct notification.
- Tier 2 (medium): goes to clinical pharmacist for human review first. 2,231 notifications sent after pharmacist evaluation.
- **Result: 72.4% overall acceptance rate** (89.8% for direct alerts, 68.4% for pharmacist-mediated). 94.5% physician satisfaction.
- Pharmacists received 17.2 alerts/day. All physicians combined received 7.8 notifications/day.

**The healthcare pattern that transfers:**
1. **Tier by severity** — don't show humans everything. Route by risk level.
2. **Interpose a specialist filter** — pharmacist reviews medium-severity before physician sees it. The pharmacist absorbs the volume so the physician doesn't have to.
3. **Replace interruptive with non-interruptive** — passive decision support instead of pop-up alerts. The information is available when needed, not pushed at high volume.
4. **Iterative redesign** — not a one-time fix. Continuous tuning of triggers and thresholds.

**Evidence level:** Level 3 (convergence across multiple hospitals and decades of research, independently validated)

**Assessment:** Healthcare's solution is the most proven because they've had the problem the longest. The transferable insight: **don't try to review everything. Tier it. Filter it. Route the right volume to the right human.** This is exactly what's missing in AI code review and AI content review — most systems still present ALL findings to ALL reviewers.

### Strategy 6: 100% Automated Scoring With Human Exception Handling — Customer Service QA

**Source:** https://www.zendesk.com/blog/ai-in-quality-assurance/ — [vendor press release]
**Source:** https://www.zendesk.com/blog/zip2-zendesk-qa-how-ai-powered-quality-reviews-help-teams-improve-faster-and-work-smarter/ — [vendor press release]
**Date:** 2025-2026

**What:** Zendesk QA and similar tools (CallMiner, AmplifAI) now score 100% of customer interactions using AI — compared to traditional sampling of 2-5%. AI evaluates empathy, tone, grammar, compliance. Humans review only flagged exceptions and trends.

**Results (vendor-reported):**
- From sampling 20/1,000 conversations to scoring all 1,000.
- Quality scores 90-96% reported by customers (CityGo).
- Real-time compliance monitoring with automatic escalation.
- "Transforms manual reviewers into strategic analysts who work with data instead of against volume."

**Limitation:** These are ALL vendor-reported numbers from Zendesk, CallMiner, etc. No independent practitioner analysis found confirming that 100% AI scoring actually reduces absorption burden vs. creating a new class of monitoring burden. The "100% coverage" number is technically correct but misleading — 100% of interactions are SCORED, but what happens when the scoring is wrong?

**Evidence level:** Level 1 (vendor claims, no independent verification)

## Counter-Evidence: Why These Solutions Might Not Work

**The false positive trap (all AI review tools):**
- Even top tools produce 5-10% incorrect findings — [domain trade publication, DevToolsAcademy](https://www.devtoolsacademy.com/blog/state-of-ai-code-review-tools-2025/)
- The "cry wolf" effect is the #1 reason teams abandon AI code review tools.
- For every real bug caught, early tools flagged nine false positives. This has improved but not been eliminated.
- A 2025 case: a mid-sized tech company accidentally wiped 47,000 user records after deploying a Copilot-recommended change that passed both automated tests AND human review. [SOURCE NEEDED — reported in aggregate, no named company]

**The Faros AI paradox (macro level):**
- Despite all these solutions, Faros AI's 10K+ developer study shows NO correlation between AI adoption and company-level improvement. The solutions may be helping individual teams while organizational-level absorption remains unsolved.
- AI adoption correlates with 9% more bugs per developer AND 154% larger PRs. The review solutions are running to stand still against growing volume.

**The quality problem compounds:**
- CodeRabbit: AI-generated code produces 1.7x more issues than human code. Logic errors up 75%, security vulnerabilities up 1.5-2x, performance issues up nearly 8x. — [practitioner analysis](https://www.coderabbit.ai/blog/state-of-ai-vs-human-code-generation-report)
- Senior engineers spend 4.3 minutes reviewing AI suggestions vs 1.2 minutes for human code. AI output is harder to review, not easier.

## Orientation

**Are there proven solutions? Yes — but they're narrow.**

The absorption bottleneck is NOT an unsolved mystery. Five distinct strategies have been implemented by named practitioners with measurable results. But every solution is domain-constrained and comes with trade-offs that shift the bottleneck rather than eliminate it:

| Strategy | What it solves | What it shifts to |
|----------|---------------|-------------------|
| Eliminate review (StrongDM) | Human review burden | Spec quality + accountability gap |
| Scalar filter (Karpathy) | Volume triage | Only works with measurable metrics |
| Agent-reviews-agent | Review volume | Shared blind spots + false positives |
| Constrain generation (SDD) | Output unpredictability | Spec-writing burden |
| Tiered review (Healthcare) | Alert volume | Specialist intermediary staffing |
| 100% auto-scoring (CS QA) | Sampling bias | Unknown failure modes |

**The meta-insight: absorption is not one problem with one solution. It's a category of problems, each requiring a domain-specific combination of strategies.**

**Healthcare is 20 years ahead of engineering on this.** Their tiered review + specialist filter + iterative threshold tuning is the most independently validated approach. Engineering is reinventing this from scratch instead of learning from healthcare's hard-won lessons.

**The most promising near-term development:** Anthropic Code Review. If the <1% false positive rate holds up under independent testing, agent-reviews-agent becomes viable for engineering. Watch for independent practitioner reports in April-May 2026.

**What would change the picture:** Someone combining StrongDM's holdout scenarios (constrain + validate) with healthcare's tiered review (route by severity) and agent-reviews-agent (filter before human sees it). Nobody is doing this yet. The strategies are siloed.
